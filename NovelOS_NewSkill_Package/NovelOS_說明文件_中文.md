## 1. 什麼是 NovelOS？

**NovelOS** 是一套專為 **長篇小說及 IP 開發** 設計的 AI 作業系統。

它的目標不是單純地「生成故事」，而是像一套完整的作業系統一樣，**管理整個小說宇宙的開發流程**——從市場研究、世界觀構建、角色設計、伏筆佈局，一直到章節寫作、品質審查與版本發布。

### 為什麼需要 NovelOS？

長篇小說開發面臨以下常見問題：

| 問題 | NovelOS 解法 |
|------|-------------|
| 世界觀吃書（前後矛盾） | Lore Database + 版本控制 |
| 角色崩壞（OOC） | 角色資料卡 + QA 審查 |
| 戰力無限膨脹 | Power Scale 管理 + 禁止無代價升級規則 |
| 伏筆遺忘不回收 | Foreshadow Manager 追蹤 |
| 多 AI 協作混亂 | AI Agent SOP 分工清晰 |
| 題材盲目開發 | Research Phase 先行市場調查 |

### 核心設計理念

- **先研究，再開發**：任何創作前必須先完成市場研究。
- **分層記憶管理**：維護四層記憶，確保設定不遺漏。
- **嚴格流程控制**：7 個 Phase 循序推進，不可跳步。
- **多 AI 分工明確**：ChatGPT / Gemini / Claude Code / Codex 各司其職。
- **版本控制強制執行**：所有修改必須留下記錄。

## 2. 核心功能一覽

### 2.1 市場研究系統（Research Phase）
- 分析近期熱門題材與讀者偏好
- 競品分析，找出市場缺口
- 神話、歷史、科學考據素材蒐集
- 爆款元素與情緒節奏分析
- 產出可行性評估報告

### 2.2 四層記憶管理系統
| 層級 | 內容 | 說明 |
|------|------|------|
| 第一層 | 核心不可變設定 | 世界規則、主線終局、能力底層邏輯 |
| 第二層 | 長期角色記憶 | 性格、關係、信念、弱點、宿命 |
| 第三層 | 中期劇情記憶 | 當前篇章、支線、伏筆、區域勢力 |
| 第四層 | 短期章節記憶 | 最近 3-5 章的情節、情緒與戰鬥狀態 |

### 2.3 世界觀資料庫（Lore Database）
- 背景設定、種族與勢力
- 地理地圖描述
- 歷史事件與傳說
- 科技／魔法能力系統規則

### 2.4 角色資料庫（Character Database）
- 標準化角色資料卡格式
- 性格、慾望、恐懼、成長線
- 能力系統與限制記錄
- 角色關係網管理

### 2.5 伏筆管理系統（Foreshadow Manager）
- 每個伏筆獨立建檔追蹤
- 記錄首次出現章節與預計回收章節
- 狀態標記：未回收 / 部分回收 / 已回收

### 2.6 戰力控制系統（Power Scale Manager）
- 每個角色或種族的戰力等級記錄
- 能力限制與代價強制登記
- 防止無代價升級的規則機制

### 2.7 情緒曲線規劃（Emotional Curve）
- 篇章情緒起伏設計
- 開端、中段、高潮、結尾情緒目標設定
- 節奏提示，引導寫作密度調整

### 2.8 時間線管理（Timeline Manager）
- 重要事件依序記錄
- 防止時間錯亂與邏輯矛盾
- 事件因果鏈追蹤

### 2.9 版本控制與工作日誌
- CHANGELOG.md：版本歷史記錄
- WORKLOG.md：每次修改的詳細日誌
- 語義化版本號（SemVer）管理

### 2.10 QA 品質審查流程
- 設定衝突檢查
- 角色 OOC 檢查
- 時間線矛盾檢查
- 戰力平衡審查

│   ├── religions.md
│   └── ...
├── 03_Characters/               # 角色資料庫
│   ├── [角色名].md
│   └── ...
├── 04_Timeline/                 # 時間線檔案
├── 05_Foreshadow/               # 伏筆管理

- `Research/` 資料**不可直接寫入** `02_Lore/` 或 `03_Characters/`
- 所有設定修改**必須同步更新** WORKLOG.md
- 重大變更**必須更新** CHANGELOG.md 並遞升版本號

## 4. AI 分工說明（Agent SOP）

NovelOS 採用多 AI 分工架構，每個 AI 負責不同階段與職能，不可互相越俎代庖。

### 4.1 Gemini — Research AI（Phase 0）

**負責階段**：Phase 0（市場研究）

**職責**：
1. 調查市場趨勢，分析熱門題材與讀者偏好
2. 進行競品分析，找出市場缺口與機會
3. 蒐集神話、歷史、科學考據素材
4. 分析爆款元素與情緒節奏
5. 產出研究報告，存入 `Research/` 目錄

**注意事項**：
- 研究結果**只做決策參考**，不得直接作為故事設定
- 研究完成後等待人類決策，確認題材可行後才可推進

---

### 4.2 ChatGPT — PM / Core（Phase 1、2、3、4、7）

**負責階段**：Phase 1、2、3、4、7（專案管理、世界觀、角色、規劃、出版）

**職責**：
1. 初始化專案：建立完整檔案結構與規則文件
2. 建立世界觀：根據研究結果建立 Lore 資料
3. 建立角色：根據需求建立角色資料卡
4. 長篇規劃：設計劇情結構、章節節奏、伏筆回收計畫
5. 維護四層記憶系統
6. 最終統整與出版準備

**注意事項**：
- ChatGPT **不參與實際章節寫作**
- 所有輸出須記錄至相應檔案並遵守版本控制

---

### 4.3 Claude Code — Main Developer（Phase 5）

**負責階段**：Phase 5（章節開發）

**職責**：
1. 根據長篇規劃撰寫小說章節
2. 發現設定變更時更新 `02_Lore/`、`03_Characters/`
3. 按需求修改章節內容，確保風格與節奏一致
4. 所有改動記錄於 `WORKLOG.md`

**注意事項**：
- Claude Code **不可自行新增或修改世界觀規則**
- 若發現規則不足，需通知 ChatGPT 補充，**不可擅自臆測**

---

### 4.4 Gemini — QA AI（Phase 6）

**負責階段**：Phase 6（品質審查）

**職責**：
1. 設定衝突檢查：檢查世界觀矛盾、設定衝突、伏筆遺漏
2. 角色 OOC 檢查：確認角色行為符合既定性格
3. 時間線檢查：確保事件順序合理，無時間錯亂
4. 戰力平衡審查：檢查能力成長是否過快或無代價
5. 在 `09_QA/` 記錄 QA 報告與修正建議

**注意事項**：
- QA 階段**只檢查與建議，不負責修改內容**
- 發現問題須回報給 ChatGPT 或 Claude Code 處理

---

### 4.5 Codex — Engineering AI（全流程按需呼叫）

**負責階段**：貫穿全流程

**職責**：
1. 開發檢查工具與模板生成腳本
2. 自動化檔案結構檢查
3. 驗證章節文件格式與 Markdown 結構

**注意事項**：
- Codex **不參與內容生成**
- 輸出以技術性檢查與自動化工具為主

## 5. 開發流程詳解（Phase 0 ～ Phase 7）

```
Phase 0 → Phase 1 → Phase 2 → Phase 3 → Phase 4 → Phase 5 → Phase 6 → Phase 7
研究      初始化    世界觀     角色       長篇規劃   章節開發   QA 審查   出版準備
(Gemini) (ChatGPT)(ChatGPT) (ChatGPT) (ChatGPT)  (Claude)  (Gemini) (ChatGPT)
```

### Phase 0：Research Phase（市場研究階段）

**目的**：在投入創作前先分析市場趨勢與題材可行性，避免盲目開發。

**執行者**：Gemini（Research AI）

**主要工作**：
- 分析近期熱門題材與讀者偏好
- 研究競品的優缺點，找出市場缺口
- 蒐集神話、歷史、科學考據素材
- 分析爆款情節元素與情緒節奏

**輸出物**：
- `Research/market_trends.md`
- `Research/audience_analysis.md`
- `Research/myth_science_notes.md`
- `Research/summary.md`（整合報告）

**完成條件**：
- 研究報告完成
- 創作者確認題材可行性
- 人類決策「進入正式開發」

---

### Phase 1：NovelOS 初始化

**目的**：建立專案基礎結構，為後續開發打好地基。

**執行者**：ChatGPT（PM）

**主要工作**：
- 建立完整資料夾結構
- 產生 `PROJECT_RULES.md`、`WORKLOG.md`、`CHANGELOG.md`
- 初始化 Lore Database 與 Character Database
- 制定禁止事項與版本控制規範

---

### Phase 2：世界觀建立

**目的**：建立完整且自洽的世界觀設定。

**執行者**：ChatGPT

**主要工作**：
- 撰寫背景概述、種族與勢力
- 建立地理描述
- 記錄歷史事件與傳說
- 制定科技／魔法能力規則、限制與代價

**存放位置**：`02_Lore/`

---

### Phase 3：角色建立

**目的**：為每個重要角色建立完整資料卡。

**執行者**：ChatGPT

**主要工作**：
- 填寫角色性格、核心慾望、恐懼與弱點
- 設計能力系統與成長線
- 建立角色關係網
- 記錄角色秘密與伏筆連結

**存放位置**：`03_Characters/[角色名].md`

---

### Phase 4：長篇規劃

**目的**：規劃劇情節奏、主要事件、伏筆佈局與回收計畫。

**執行者**：ChatGPT

**主要工作**：
- 建立時間線（`04_Timeline/`）
- 設計伏筆清單（`05_Foreshadow/`）
- 規劃戰力成長路徑（`06_PowerScale/`）
- 設計各篇章情緒曲線（`07_EmotionalCurve/`）

---

### Phase 5：章節開發

**目的**：根據長篇規劃撰寫實際小說章節。

**執行者**：Claude Code（Main Developer）

**主要工作**：
- 撰寫各章節草稿
- 保持世界觀與角色設定一致
- 每次修改後更新 `WORKLOG.md`
- 重大修改更新 `CHANGELOG.md`

**存放位置**：`08_Chapters/`

---

### Phase 6：QA 品質審查

**目的**：確保章節品質，避免設定衝突與崩壞問題。

**執行者**：Gemini（QA AI）

**主要工作**：
- 設定衝突與世界觀矛盾檢查
- 角色 OOC 審查
- 時間線邏輯驗證
- 戰力平衡檢查
- 輸出修正建議報告

**存放位置**：`09_QA/`

---

### Phase 7：版本同步與出版準備

**目的**：整合所有修訂，更新版本記錄，準備出版格式。

**執行者**：ChatGPT

**主要工作**：
- 整合 Claude 與 Gemini 的成果
- 更新版本號與版本紀錄
- 進行翻譯作業
- 出版格式準備

## 6. 所有範本說明與使用方式

### 6.1 角色資料卡（CHARACTER_TEMPLATE.md）

**存放路徑**：`03_Characters/[角色名].md`

**欄位說明**：

| 欄位 | 說明 |
|------|------|
| 定位／角色類型 | 主角、配角、反派、NPC 等 |
| 初始性格 | 性格特質描述（內向、自信、狡猾等） |
| 核心慾望 | 角色最渴望達成的目標或願望 |
| 恐懼與弱點 | 內心弱點與害怕的事物 |
| 能力與技能 | 特殊能力與技能，**必須說明限制與代價** |
| 成長線／進化路線 | 初始 → 中期 → 最終形態 |
| 關係網 | 與其他角色的關係（敵對、盟友、師徒等） |
| 秘密與伏筆 | 隱藏秘密，需對應 Foreshadow Manager 條目 |
| 當前狀態 | 目前章節的狀態、情緒與地點 |
| 不可違反規則 | 核心禁忌與不可動搖的設定 |

**重要提醒**：
- 角色資料卡應在**首次登場前**建立
- 更新後必須於 `WORKLOG.md` 記錄

---

### 6.2 世界觀資料（LORE_TEMPLATE.md）

**存放路徑**：`02_Lore/[類別].md`

**欄位說明**：

| 欄位 | 說明 |
|------|------|
| 背景概述 | 世界整體背景與時代 |
| 種族與勢力 | 各主要種族、族群、國家或組織 |
| 地理與地圖 | 重要區域、城鎮或星系 |
| 歷史事件與傳說 | 影響當前局勢的重要歷史事件 |
| 宗教與神話 | 宗教體系、信仰與神話故事 |
| 科技／魔法規則 | **力量來源、運作原理、限制與代價**（必填） |
| 進化與階級制度 | 角色或種族的進化階梯與社會階層 |
| 重要物品與資源 | 神器、關鍵資源或遺跡 |
| 終局伏筆 | 主線終局的相關伏筆 |

**重要提醒**：
- 所有世界觀設定需經 ChatGPT 審核後才可加入
- 任何修改必須同步更新版本號

---

### 6.3 時間線管理（TIMELINE_TEMPLATE.md）

**存放路徑**：`04_Timeline/[篇章或事件名].md`

**欄位說明**：

| 欄位 | 說明 |
|------|------|
| 發生章節 | 事件在小說中的章節位置 |
| 發生時間 | 具體日期、年份或世界觀曆法 |
| 發生地點 | 地理位置或區域 |
| 參與角色 | 所有重要角色名單 |
| 事件經過 | 事件過程概述 |
| 造成結果 | 直接後果 |
| 後續影響 | 長遠影響 |
| 潛在衝突 | 可能與其他設定產生的矛盾（供 QA 參考） |

**重要提醒**：
- 角色移動、康復或資訊傳遞的時間跨度需**合理計算**
- 複雜事件可拆分多個子事件並互相鏈結

---

### 6.4 伏筆管理（FORESHADOW_TEMPLATE.md）

**存放路徑**：`05_Foreshadow/[伏筆名稱].md`

**欄位說明**：

| 欄位 | 說明 |
|------|------|
| 伏筆類型 | 主線、世界觀、角色、情感、陰謀、終局 |
| 首次出現章節 | 伏筆最初出現位置 |
| 涉及角色 | 相關角色清單 |
| 內容描述 | 伏筆內容及提示方式 |
| 預計回收章節 | 預計揭露的章節（可修改，需記錄） |
| 目前狀態 | **未回收 / 部分回收 / 已回收** |
| 回收方式 | 伏筆回收時的處理方式 |
| 備註 | 與其他伏筆的連動關係 |

**重要提醒**：
- 所有伏筆**必須在合理章節回收**
- 預計回收時間變更需更新此檔並記錄

---

### 6.5 戰力控制（POWER_SCALE_TEMPLATE.md）

**存放路徑**：`06_PowerScale/[角色或種族名].md`

**欄位說明**：

| 欄位 | 說明 |
|------|------|
| 目前戰力等級 | 當前階段的戰力等級（數值或等級系統） |
| 能力描述 | 能力特性、優勢與使用方式 |
| 限制與代價 | **能量消耗、傷害反噬、時間間隔等**（必填） |
| 弱點 | 易被克制的方式或自然弱點 |
| 可擊敗對象 | 現階段可戰勝的對手 |
| 不可擊敗對象 | 現階段無法戰勝的對手 |
| 下一階段成長條件 | 升級或進化的條件 |
| 是否成長過快 | 自我檢查成長速度是否合理 |

**重要提醒**：

## 7. 專案規則與禁止事項

### 7.1 一般原則

1. **資料分層**：Research 資料存 `Research/`，正式設定存 `02_Lore/` 等，**絕不混用**。
2. **流程順序**：Research Phase 完成且題材確認後，才可啟動正式開發。**不可跳步**。
3. **版本控制**：每次重大修改更新 `CHANGELOG.md` 並於 `WORKLOG.md` 記錄。
4. **工作日誌**：所有 AI 修改資料時**立即**更新 `WORKLOG.md`。未記錄之修改視為無效。

### 7.2 七大禁止事項

| 禁止行為 | 說明 |
|----------|------|
| 禁止自行補完未確認設定 | 未審核設定不可加入，需通過 ChatGPT 確認 |
| 禁止世界觀吃書 | 不得與既定規則或前文設定矛盾 |
| 禁止角色 OOC | 角色言行須符合性格與成長線 |
| 禁止無代價升級 | 能力成長必須有對應代價或限制 |
| 禁止不回收伏筆 | 所有伏筆必須在合理章節回收並記錄 |
| 禁止未更新版本紀錄 | 重大設定變更若未更新 CHANGELOG 即視為無效 |
| 禁止未經研究直接開發 | Research Phase 未完成，不得進行世界觀或章節開發 |

### 7.3 建議遵守事項

- **溝通清晰**：修改或建議時使用清晰文件與標籤
- **定期 QA**：每完成一定進度後執行 Gemini QA 檢查
- **重用範本**：優先使用範本文件，保持格式一致
- **保護核心設定**：核心不可變設定僅允許 ChatGPT 修改

## 10. 常見問題 FAQ

**Q：可以直接跳過 Research Phase 開始創作嗎？**
A：不可以。PROJECT_RULES.md 明確禁止未經研究直接開發。Research Phase 的目的是確保題材有市場潛力，避免投入大量時間後才發現題材不可行。

**Q：如果研究完成後創作者決定不想做了，怎麼辦？**
A：研究結果會保存在 `Research/` 目錄，日後想要重啟時可以直接參考，不需重新研究。

**Q：Claude Code 在寫作時發現世界觀有漏洞，可以自己補完嗎？**
A：不可以。必須通知 ChatGPT（PM）補充設定，Claude Code 不可自行臆測或補充未確認的世界觀規則。

**Q：QA 發現問題後，誰來修改章節？**
A：Gemini（QA AI）只提出報告與建議，修改工作由 Claude Code 負責，重大設定修改由 ChatGPT 確認。

**Q：伏筆可以永遠不回收嗎？**
A：不可以。PROJECT_RULES.md 禁止不回收伏筆。所有伏筆必須在合理章節回收，若確認該伏筆不再使用，需在 Foreshadow 檔案中標記並記錄原因。

**Q：版本號怎麼決定要升哪一位？**
A：參考 SemVer 規範：小修正升修訂版（Z）、新功能或新章節升次版本（Y）、重大架構變更升主版本（X）。
